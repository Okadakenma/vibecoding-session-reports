3. 講義・技術内容
① 備品管理システム試作の発表（小池田康徳）
なぜ重要か：第3回の課題（DBとデプロイ）が定着したことを示す初の実証成果。受講者がプロンプト1本で実用的なシステムを作れるレベルに達した証明となった。
  • 役員がGeminiで試みたがDB接続で詰まったシステムを、自ら学んだ内容で独力完成
  • 機能：ログイン（マスター/一般ユーザー）、在庫管理、アラート、持ち出し履歴（時系列・誰が/何を/何個）、持ち出し予約登録
  • チェックアウトレコードが品目IDと従業員IDで紐づく → リレーショナルDBの概念を実地で体験
  • セキュリティ：.gitignore + .env.local でNeon DB URLを管理。GitHubリポジトリのPrivateモード確認も推奨
  • Claudeにディレクトリを掴ませて「セキュリティ的に大丈夫か」と確認する習慣を推奨（ハードコードされたAPIキーがないかのチェック）
② スキルとコンテキストエンジニアリング
なぜ重要か：AIに特定の専門性を持たせることで成果物の品質が飛躍的に向上する。自社フォーマットをスキル化すれば毎回同じ指示を書く手間が省け、出力が安定する。
  • スキル：AIに特定の専門能力を持たせる仕組み。Anthropic公式の「フロントエンドデザイン」スキルをインストールして起動することでダッシュボードのデザインがモダンに一新された
  • スキルは自作も可能 → 自社デザインフォーマット・PowerPointフォーマット等をスキルとして登録し再利用できる（「スキルクリエイター」スキルを使用）
  • コンテキストエンジニアリング：AIに特定の文脈を持たせて成果物を出させる技術の総称。スキルもその一形態
  • スキルを強制起動させるには、有効化の手順が必要であることを小池田氏が実地で発見
③ CLAUDE.md によるグローバル・プロジェクト設定
なぜ重要か：毎回AIに同じ前提を説明する手間を省ける。プロジェクト固有のルール（技術スタック・禁止事項等）を一度書けばAIが常にそのコンテキストで作業を開始する。
  • ~/.claude/CLAUDE.md（グローバル）：全プロジェクトに適用される共通設定
  • プロジェクト内の CLAUDE.md：そのプロジェクト固有のルール。AIは起動時に必ず読み込む
  • 統合ポータル運用時は親フォルダの CLAUDE.md に「不要なプロジェクトのソースを読み込まないルール」を記述することでコンテキスト圧迫を防げる
④ 仕様書駆動開発フローの確立
なぜ重要か：仕様を固めずに開発すると手戻りが大きくなる。Claude（チャット版）で要件を対話的に詰めてから Claude Code に渡すことで、一発で品質の高いアプリが完成する。
  • 推奨フロー：① Claudeチャット（Opus）で仕様書（Markdown）を作成 → ② 仕様書をClaude Codeに読み込ませてタスクステップ生成 → ③ サブエージェント方式で並行開発
  • Claudeは不明点を都度確認しながら仕様を固めるため、ChatGPTやGeminiが勝手に進める傾向があるのに対して、仕様を固める工程では特に有効
  • 共通技術スタック：Next.js（フルスタック）/ Neon（PostgreSQL）/ Vercel（デプロイ）/ Anthropic API / Cloudflare R2（画像・PDF）/ Vercel Cron Jobs（スケジュール実行）
  • サブエージェント方式はトークン消費が激しくなるため、Pro Planでは上限に達しやすい。開発時はMax Planへの移行を推奨
⑤ 認証（Authentication）・認可（Authorization）の概念
なぜ重要か：業務アプリに必須の機能。概念を理解しておくことで、AIへの指示の精度が上がり、権限設計のミスを防げる。
  • 認証（Authentication）：「あなたであることの確認」＝ ログイン。メールアドレス＋パスワード等で本人を証明する
  • 認可（Authorization）：「何ができるか」＝ 管理者権限・一般権限など、ログイン済みユーザーに対してできる操作の範囲を制御する
  • Google OAuth（Googleアカウントでログイン）は、会社全員がGoogleアカウントを持っている場合に有効。そうでない場合はメール＋パスワード方式が安全
  • 各アプリが個別にログインを持つのは非効率 → 統合ログイン基盤を1つ構築し、配下に各アプリを配置する方針に決定
⑥ コスト・料金体系とリソースの使い回し
なぜ重要か：開発を続けるには費用感を把握しておく必要がある。多くのコストはDB・デプロイの2点に集約され、適切に管理すれば業務アプリ運用はほぼ無料で維持できる。
  • コスト発生源の2大要素：Neon（データ量×アクセス回数）と Vercel（デプロイ）。業務用途の範囲内はほぼ無料
  • 画像・PDF・動画など重いデータはNeonではなくCloudflare R2などの外部ストレージに保存するのがコスト最適
  • NeonのDB URL・AnthropicのAPIキーなど共通リソースはプロジェクトごとに新規発行せず使い回すと管理が楽になる
  • Claude Max Planへの移行：Pro Planはサブエージェント方式・Opus使用でトークン上限に達しやすい。開発フェーズはMax Plan推奨（今回2名が上限到達）
  • 追加クレジット購入（上限到達時に自動消費）の選択肢もあり。使用頻度によってはMax Planより安価になる場合がある
⑦ 既存システム連携の自動化（Python・Playwright）
なぜ重要か：新規アプリを作るだけでなく、既存の業務システムとの連携を自動化できると、RPA的な価値を低コストで実現できる。
  • 課題例（田中翔）：NIコラボ360からCSV出力 → 手動で不要列を削除 → 生産管理システムへインポート、という手作業の自動化
  • 提案解決策：ClaudeでWebアプリを作り所定のCSV形式で出力 → Pythonで突合・自動インポート
  • Playwright：ブラウザ操作を自動化するOSSフレームワーク。Claudeと組み合わせることで既存Webアプリの操作も自動化できる
  • Python + Playwright + Windowsタスクスケジューラを組み合わせることで、CSVダウンロードからクラウドストレージへのアップロードまでの一連処理を自動化できる
4. 主要ディスカッション（Q&A）
Q. VS CodeでClaudeを開こうとするとログインが外れてAPIキーエラーが発生する。（田中翔）
ログインが外れた状態のため、「ログイン」を実行してサブスクリプションと紐付けることで解決できる。APIキーを手動で入力する画面が表示された場合は、ログイン操作を選ぶ。
Q. 認証にGoogleアカウントのOAuth認証を使えるか？（小池田康徳）
実装自体は可能。ただし、会社全員がGoogleアカウントを持っている場合のみ有効。そうでない場合はログインできないユーザーが出るため、メールアドレス＋パスワード方式を基本とすることを推奨。
Q. 日報アプリで週次レポートを自動生成する際、Vercel Cron Jobsを使って問題ないか？（小池田康徳）
デプロイ済みアプリのスケジュール実行にはVercel Cron Jobsを使う想定で問題ない。無料プランでは1日1回まで実行可能。それ以上の頻度が必要な場合は有料プランへのアップグレードを検討する。
Q. 日報に写真を添付する場合、画像ストレージはどれを使うべきか？（小池田康徳）
Cloudflare R2の使用が最も簡単でおすすめ。画像・PDFどちらにも対応しており、コストも安価。コストを抑えるためにも、ファイル類はNeonに保存せず外部ストレージに置くのが鉄則。
Q. 統合ポータル配下に複数アプリのフォルダーを置くとClaudeのコンテキストが圧迫されないか？（文田浩輔）
懸念は正しい。対策として親フォルダの CLAUDE.md に「現在作業中のプロジェクト以外のソースを読み込まないルール」を記述することでコンテキストの圧迫を防げる可能性がある。ディレクトリ設計はClaude Codeと相談しながら決める。
Q. ChatGPTやGeminiとClaudeの違いは何か？
ChatGPTやGeminiは指示に対して積極的に進める傾向があるのに対し、Claudeは不明点や確認事項を都度ユーザーに尋ねる慎重な設計になっている。仕様を固める上流工程ではClaudeの慎重さが品質向上に直結する。
